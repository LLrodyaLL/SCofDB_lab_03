### Запрос №1

```sql
SELECT id, user_id, status, total_amount, created_at
FROM orders
WHERE user_id = (
    SELECT id
    FROM users
    ORDER BY created_at
    LIMIT 1
)
ORDER BY created_at DESC
LIMIT 50;
```
Почему запрос медленный:

- выполняется фильтрация по `user_id`;
- затем требуется сортировка по `created_at DESC`;
- без составного индекса серверу приходится читать больше строк и выполнять дополнительную сортировку.

### Запрос №2

```sql
SELECT id, user_id, status, total_amount, created_at
FROM orders
WHERE status = 'paid'
  AND created_at >= TIMESTAMP '2025-01-01'
  AND created_at < TIMESTAMP '2025-04-01'
ORDER BY created_at DESC
LIMIT 100;
```
Почему запрос медленный:

- используется фильтрация сразу по двум условиям: `status` и диапазону `created_at`;
- существующего индекса только по `status` недостаточно;
- для сортировки по дате также нужны дополнительные затраты.

### Запрос №3

```sql
SELECT
    o.user_id,
    COUNT(DISTINCT o.id) AS orders_count,
    SUM(oi.price * oi.quantity) AS revenue
FROM orders o
JOIN order_items oi ON oi.order_id = o.id
WHERE o.created_at >= TIMESTAMP '2025-01-01'
  AND o.created_at < TIMESTAMP '2026-01-01'
GROUP BY o.user_id
ORDER BY revenue DESC
LIMIT 20;
```
Почему запрос медленный:

- присутствует `JOIN` большой таблицы заказов с таблицей позиций заказа;
- после соединения выполняется `GROUP BY`;
- затем выполняется сортировка агрегированных данных;
- даже при наличии индексов значительная часть времени уходит на агрегацию большого объёма строк.

### Индекс №1
- SQL: 
```sql
CREATE INDEX IF NOT EXISTS idx_orders_user_id_created_at_desc
    ON orders USING BTREE (user_id, created_at DESC);
```
- Какой запрос ускоряет: запрос №1.
- Почему выбран тип: BTREE оптимален для поиска по равенству (user_id = ...) и сортировки (ORDER BY created_at DESC).

### Индекс №2
- SQL: 
```sql
CREATE INDEX IF NOT EXISTS idx_orders_status_created_at_desc
    ON orders USING BTREE (status, created_at DESC);
```
- Какой запрос ускоряет: запрос №2.
- Почему выбран тип: `BTREE` хорошо подходит для составного условия вида status = const + диапазон по created_at, а также помогает уменьшить стоимость сортировки.

### Индекс №3
- SQL: 
```sql
CREATE INDEX IF NOT EXISTS idx_orders_created_at_brin
    ON orders USING BRIN (created_at);
```
- Какой запрос ускоряет: аналитические запросы по диапазонам дат, в том числе запрос №3 и запросы после партиционирования.
- Почему выбран тип: `BRIN` эффективен на больших таблицах, где значения created_at коррелируют с физическим расположением строк; такой индекс компактный и дешёвый в хранении.

## 5. Партиционирование `orders` по дате
### 5.1 Выбранная стратегия
_TODO: Опишите стратегию (например, RANGE по месяцам)._

Была выбрана стратегия `RANGE-партиционирования` по полю `created_at` с разбиением по кварталам.

Причины выбора:

- большая часть аналитических запросов использует фильтрацию по дате;
- PostgreSQL может исключать нерелевантные партиции (`partition pruning`);
- такой подход особенно полезен для диапазонных запросов.

## 8. Что не удалось исправить только индексами
_TODO: Перечислите случаи, где индексы не помогли, и объясните почему._

Индексы не решили полностью следующие проблемы:

1. Тяжёлые запросы с `JOIN + GROUP BY` по большому объёму данных всё равно остаются дорогими.
2. Полные агрегаты по широкому периоду по-прежнему требуют чтения значительной части таблицы.
3. Если запрос выбирает очень большую долю строк, PostgreSQL может предпочесть Seq Scan, потому что индексный доступ становится невыгодным.
4. Для таких случаев дополнительно помогают:
    - партиционирование;
    - переписывание запроса;
    - предагрегация;
    - materialized views.

Подсказки:
- full scan при высокой доле выбираемых строк;
- тяжёлые `GROUP BY`/`ORDER BY` на большом объёме;
- необходимость переписывания запроса или pre-aggregation.
